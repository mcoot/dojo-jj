Dojo CLI E2E Test Execution

A Go-native, opt-in E2E harness that runs the compiled dojo binary against isolated Jujutsu repositories.

Status: Draft for review Artifact: plan.md Updated: 2026-06-27

Keyboard, scroll, and progress navigation are enabled.

01 - Problem

The placeholder suite is not an executable black-box E2E harness.

Tests need to stay ordinary Go tests while exercising the compiled CLI, real JJ commands, and disposable repositories.

Current state

internal/e2e has a placeholder suite with no Test... entry point.

Target state

test/e2e runs tagged Go tests through DOJO_BIN.

Constraint

No production internals are imported by E2E tests.

Isolation

Repos, workspaces, JJ config, and HOME live under temp roots.

02 - Approach

Use Go tests as the harness and subprocesses as the boundary.

The Taskfile builds the artifact; the E2E package validates it, prepares isolated JJ state, and runs real CLI commands.

  1. Wire the package
    Move to test/e2e, add build tags, register the suite, and add task e2e.
  2. Create the world
    Resolve DOJO_BIN, build deterministic env, and capture subprocess results.
  3. Isolate JJ state
    Generate a colocated template repo and copy it for each test.
  4. Assert CLI behavior
    Run dojo --help and dojo get from copied repos.

03 - Acceptance

Done means both normal and tagged test workflows are proven.

  • go test ./... passes with DOJO_BIN unset.
  • task e2e builds out/dojo and runs go test -tags=e2e ./test/e2e.
  • Tagged tests fail clearly when DOJO_BIN is missing or invalid.
  • E2E tests create all mutable state under t.TempDir and use test-owned JJ_CONFIG.
  • rg 'github.com/mcoot/dojo-jj/internal/(cmd|factory|service|dependencies)' test/e2e returns no matches.
Manual verification is task test, then task e2e, then one targeted failure-path run for dojo get without JJ on PATH.

04 - Architecture

Artifact-first test flow

The compiled binary is supplied by the caller; the test process owns setup and assertions only.

flowchart LR
  Task[task e2e] --> Build[go build out/dojo]
  Build --> GoTest[go test -tags=e2e]
  GoTest --> World[test/e2e/world]
  World --> Config[Temp HOME and JJ_CONFIG]
  World --> Template[Generated JJ template]
  Template --> Copy[Per-test repo copy]
  Copy --> Dojo[DOJO_BIN subprocess]
  Copy --> JJ[jj subprocess]
  Dojo --> Assert[Assertions]
  JJ --> Assert

05 - Contracts

Small world API, strict subprocess rules

world.New(t)

Validates DOJO_BIN, resolves jj, creates temp root, HOME, and JJ config.

RunDojo and RunJJ

Run with explicit working directory, timeout, isolated env, and separate stdout/stderr capture.

NewRepoFromTemplate

Copies the read-only generated colocated JJ template into a per-test repo path.

  • Use jj subprocesses only; no direct git invocations.
  • Assert substrings for log.Fatal stderr because timestamps may be present.
  • Keep helper behavior testable with fake executables or Go helper processes.

09 - Milestone 4

Black-box CLI E2E coverage

Tests

dojo --help from a copied repo, dojo get with JJ on PATH, and dojo get with JJ hidden from PATH.

Acceptance

  • Help exits 0 and mentions the root command and get.
  • dojo get exits 0 when JJ is available.
  • dojo get exits non-zero and stderr contains JJ not found on path when JJ is hidden.
dojo get currently performs no workspace mutation when JJ exists, so this first E2E suite should stay scoped to the current observable behavior.

10 - Invariants

Rules that keep the harness honest

These constraints should be checked during implementation and review.

Opt-in only

Normal go test ./... does not need E2E setup.

Black box

E2E tests run DOJO_BIN; they do not instantiate internals.

Disposable state

All mutable state stays under temp directories.

No new deps

The initial harness uses the Go standard library, testify, Taskfile, and installed jj.

11 - Stop and ask

Implementation should pause if the plan assumptions break.

  • E2E tests require production CLI behavior changes beyond current dojo get success/error paths.
  • JJ_CONFIG=<file> is not portable enough and needs a different isolation model.
  • The implementation would commit nested .jj or .git fixtures.
  • The Go tests would auto-build the binary instead of requiring explicit DOJO_BIN.
